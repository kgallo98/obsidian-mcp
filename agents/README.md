# obsidian-mcp — Knowledge Tree

A self-improving knowledge base for AI coding assistants.

## How It Works

```
Session 1:  AI reads AGENTS.md + active-context.md → works → writes learnings
Session 2:  AI reads updated files → has more context → works better → writes more
Session N:  Knowledge compounds → AI gets genuinely useful for YOUR project
```

## Structure

```
agents/
  _index.md          (routing node)
  active-context.md  (session handoff state)
  README.md          (tree sitemap)
```

## The Growth Pattern

The tree grows organically. Don't plan it upfront — let it emerge from work.

| Sessions | What Happens |
|----------|-------------|
| 1-5 | Root files + first domain file. AI learns project basics. |
| 5-20 | 3-5 domain files. Known Issues accumulate. AI avoids past mistakes. |
| 20-50 | Routing takes shape. Commands & Patterns fill in. AI is genuinely helpful. |
| 50-100 | Deep knowledge. Cross-domain awareness. AI is a project expert. |
| 100+ | The tree IS the project's institutional memory. |

## Key Principle

**Only write what the AI can't discover on its own.**

Good entries: custom commands, environment quirks, past mistakes, architectural decisions.
Bad entries: how the language works, standard library usage, obvious project structure.

Research shows: excess context HURTS performance (+20% cost, only +4% benefit).
Focus on gotchas, quirks, and decisions — not tutorials.

## Maintenance

- **Weekly**: skim session logs, check nothing is stale
- **Monthly**: review Known Issues (are they still issues?), prune resolved ones
- **When a file hits 750 lines**: split it at ## boundaries

Knowledge rots in about 3 weeks without maintenance. A small, accurate tree
beats a large, outdated one every time.
